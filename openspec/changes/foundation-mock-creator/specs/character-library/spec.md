## 新增需求

### 需求:用户可管理家庭固定角色

系统必须提供家庭角色库，支持固定主角的创建、查看、编辑、删除；每个角色必须包含名字、关系（如 baby/dad/mom/other）、外貌描述文本、参考图 URL（占位即可）。

#### 场景:创建家庭角色

- **当** 用户在角色库填写名字、关系、外貌描述并保存
- **那么** 新角色必须出现在角色列表，且具备唯一 id

#### 场景:编辑角色外貌描述

- **当** 用户修改某角色外貌描述并保存
- **那么** 更新后的描述必须在角色详情中可见

### 需求:每个固定角色必须对应单张参考图

系统必须遵循「一角色一图」规则：每个角色的参考形象必须绑定单一 `referenceImageUrl`；禁止将多角色 turnaround 合并为同一参考引用入口。

#### 场景:查看角色参考图

- **当** 用户打开某角色详情
- **那么** 界面必须展示且仅展示一张主参考图

#### 场景:模拟生成参考图

- **当** 用户点击「模拟生成参考图」
- **那么** 系统必须用占位图 URL 替换该角色参考图，且不得调用真实生图 API

### 需求:新建故事时可勾选出场固定角色

新建项目流程必须允许多选家庭角色库中的主角；所选角色 id 必须写入项目的 `castCharacterIds` 字段。

#### 场景:新建故事勾选主角

- **当** 用户在新建向导勾选「宝宝」与「爸爸」并创建项目
- **那么** 项目元数据必须包含对应角色 id，且大纲生成 mock 应标注这些主角出场信息

### 需求:临时角色不得进入家庭角色库

故事临时角色（如三只小猪、大灰狼）必须仅出现在分页大纲或页面元数据中；系统禁止将临时角色写入全局家庭角色库。

#### 场景:大纲含临时角色

- **当** mock 大纲为某页标注临时角色「小黑猪」
- **那么** 「小黑猪」必须出现在该页元数据，且不得出现在 `/characters` 家庭库列表
